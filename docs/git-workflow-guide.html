1. Our Branching Strategy

We use a three-tier branching model. All work flows in one direction: from feature branches, through staging, to production.

The Three Branch Types

Branch Purpose Deploys To
main Production-ready code. Only updated via PR from preview. gptechadvisors.com
preview Staging/integration branch. All feature work merges here first. preview.gptechadvisors.com
feature/* Your working branch for a specific task. Created from preview. Local only (until PR is merged)
The flow: feature/* → PR to preview → verify on staging site → PR to main → live on production
Never push directly to main or preview. All changes go through pull requests. This protects the live site and gives the team a chance to review.

3. Daily Workflow

Every time you sit down to work, start with a fresh pull:

# Make sure you're on preview
git checkout preview

# Pull the latest changes from the remote
git pull origin preview

# Now create your feature branch (or switch to an existing one)
git checkout -b feature/your-feature-name
Ask Claude instead
Pull the latest preview branch and create a new feature branch called feature/your-feature-name
Branch naming convention: Use feature/ followed by a short, descriptive name with hyphens. If there's a Jira ticket, prefix it.
Examples: feature/fix-services-hero, feature/blog-system, feature/JIRA-42-contact-form

4. Refreshing from Remote

If your branch has fallen behind preview (for example, someone else's PR got merged while you were working), you need to pull in those changes.

Option A: Merge preview into your branch (preferred)

# From your feature branch:
git fetch origin
git merge origin/preview

# If there are conflicts, Git will tell you which files.
# Fix them, then:
git add .
git commit
Ask Claude instead
My feature branch is behind preview. Merge the latest preview into my current branch and resolve any conflicts.

Option B: Rebase onto preview (cleaner history)

# From your feature branch:
git fetch origin
git rebase origin/preview

# If there are conflicts, fix them, then:
git add .
git rebase --continue
Ask Claude instead
Rebase my current feature branch onto the latest preview branch.
When to refresh:
  • Before starting new work each day
  • Before creating a pull request
  • When GitHub tells you your PR has merge conflicts

5. Working on a Feature Branch

Here's the full lifecycle of a feature branch:

1 Create the branch from an up-to-date preview:

git checkout preview
git pull origin preview
git checkout -b feature/my-new-feature
Ask Claude instead
Refresh preview from remote and create a new branch called feature/my-new-feature

2 Make your changes. Commit often with clear messages:

# Stage specific files (preferred over "git add .")
git add css/pages.css services.html

# Commit with a descriptive message
git commit -m "fix: match services hero padding to live site"
Ask Claude instead
Commit my changes to css/pages.css and services.html with the message "fix: match services hero padding to live site"

Or simply: /commit — Claude will look at your changes and write an appropriate commit message.

3 Push your branch to GitHub:

# First push (sets up tracking)
git push -u origin feature/my-new-feature

# Subsequent pushes
git push
Ask Claude instead
Push my current branch to GitHub

4 Create a Pull Request on GitHub (see next section).

5 After the PR is merged, clean up:

git checkout preview
git pull origin preview
git branch -d feature/my-new-feature
Ask Claude instead
Switch back to preview, pull the latest, and delete my old feature/my-new-feature branch

6. Creating a Pull Request

A Pull Request (PR) is how you propose merging your feature branch into preview. It gives the team a chance to review your changes before they go live on staging.

Using the GitHub CLI (fastest)

# Make sure all your changes are committed and pushed
git push -u origin feature/my-new-feature

# Create the PR targeting preview
gh pr create --base preview \
  --title "feat: add blog filtering system" \
  --body "Adds category filtering to the blog page with animated transitions."
Ask Claude instead
Push my branch and create a pull request to merge into preview

Claude will examine your commits, write a PR title and description, and create it for you.

Using the GitHub Website

  1. Go to github.com/GrowthPointTech/gp-website
  2. Click "Compare & pull request" (GitHub usually shows a banner after you push)
  3. Set the base branch to preview (not main!)
  4. Write a clear title and description of what you changed
  5. Click "Create pull request"
Always target preview, not main. Feature branches merge into preview. Only preview merges into main (Stacey handles this).

Writing a Good PR Description

Include:

  • What you changed (a brief summary)
  • Why (what problem does it solve or what does it add)
  • How to test (what should the reviewer look at)
  • Screenshots if it's a visual change

7. Code Review & Merging

Once a PR is created:

  1. The team reviews the changes on GitHub. Reviewers can leave comments, request changes, or approve.
  2. Fix any feedback by making additional commits on the same branch and pushing. The PR updates automatically.
  3. Once approved, the PR gets merged into preview. This triggers an automatic deploy to preview.gptechadvisors.com.
  4. Verify on the staging site that everything looks correct.
  5. When ready for production, Stacey creates a PR from preview to main, which deploys to the live site.

8. Handling Merge Conflicts

Merge conflicts happen when two people edit the same part of a file. Git can't decide which version to keep, so it asks you to resolve it manually.

What Conflicts Look Like

Git marks conflicts in the file like this:

<<<<<<< HEAD
  padding: 30px 60px;
=======
  padding: 16px 40px;
>>>>>>> origin/preview
  • The code between <<<<<<< HEAD and ======= is your version
  • The code between ======= and >>>>>>> is the incoming version

How to Resolve

  1. Open the conflicted file in your editor
  2. Decide which version to keep (or combine both)
  3. Delete the conflict markers (<<<<<<<, =======, >>>>>>>)
  4. Stage and commit:
git add path/to/resolved-file.css
git commit -m "fix: resolve merge conflicts with preview"
Ask Claude instead
Resolve the merge conflicts on my current branch

Claude will find all conflicted files, examine both sides, intelligently resolve each conflict, and commit the result. This is especially helpful for complex conflicts across multiple files.

Tip: VS Code has a built-in merge conflict editor. When you open a conflicted file, it highlights both versions and gives you buttons to accept one, the other, or both.

9. Golden Rules

  1. Never push directly to main or preview. Always use a feature branch and a PR.
  2. Pull before you push. Always fetch/pull the latest before starting work or creating a PR.
  3. Commit often, push regularly. Small, frequent commits are easier to review and revert if needed.
  4. Write clear commit messages. Future-you will thank present-you.
    Good: fix: match services hero padding to live site
    Bad: fixed stuff
  5. Don't commit secrets. Never commit .env files, API keys, or credentials.
  6. Ask if you're unsure. It's always better to ask than to force-push and lose work.
