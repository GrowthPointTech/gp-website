Step 1: Ask Claude to compare ONE page at a time

Open Claude Code in the gp-website directory and run a comparison for one page per prompt. Don't ask Claude to compare multiple pages at once.

Why one page at a time? AI works best with focused tasks. When you ask Claude to compare 3 pages at once, it has to hold a lot of information in context and tends to rush through or miss details. One page at a time means Claude can go deep on each section, catch small differences, and give you more accurate results. This is a good general principle for working with AI: smaller, focused prompts get better results than big, vague ones.

BOTH Responsive Design (all pages)

What is "responsive" design?

People visit websites on all kinds of screens — wide desktop monitors, laptops, tablets, and phones. A responsive website automatically adjusts its layout to look good at any screen width. Text reflows, images resize, and multi-column layouts stack into single columns on smaller screens.

You've probably noticed this yourself: a website that shows 3 cards side-by-side on your laptop shows them stacked vertically on your phone. That's responsive design.

How to test it

You don't need a phone and a tablet to test. Your browser can simulate any screen width:

  1. Open the page in Chrome
  2. Right-click anywhere on the page and choose "Inspect"
  3. Click the device toolbar icon (looks like a phone/tablet) at the top of the Inspector panel — or press Cmd+Shift+M
  4. At the top of the page, you'll see a width dropdown. You can type in any width or drag the edge of the viewport to resize.

Widths to test

Width What it simulates What to look for
1920px Wide desktop monitor Content stays centered, doesn't stretch to edges. Nav logo/links aren't pushed to far corners.
1400px Standard laptop This is our "baseline" — everything should look great here.
1024px Tablet (landscape) 2-column layouts may start stacking. Font sizes may shrink. Check nothing overlaps.
768px Tablet (portrait) Most multi-column sections should stack to 1 column. Nav may switch to hamburger menu.
375px iPhone / small phone Everything single-column. Text readable without zooming. No content cut off or hidden. No horizontal scrollbar.

What to check at each width

  • No horizontal scrollbar — if you see one, something is wider than the screen
  • Text doesn't overflow — no text running off the right edge or getting cut off
  • Images scale down — images should shrink to fit, not stay full-size and break the layout
  • Columns stack properly — a 2-column layout on desktop should become 1 column on mobile
  • Buttons/links are tappable — on mobile widths, buttons should be big enough to tap (at least 44px tall)
  • Text is readable — font sizes shouldn't be too small on mobile

Phase 2: Triage

Once both of you have completed your findings lists, meet (or Slack) to:

  1. Combine your findings into a single master list
  2. Remove false positives — anything that's intentionally different (Contact Us removal) or acceptable (font-family fallback differences like system-ui)
  3. Prioritize — High severity first, then Medium, then Low
  4. Create GitHub Issues for each finding (or group related findings into one issue)

Phase 3: Fix

For each issue, follow the git workflow you know:

  1. Pull latest preview
  2. Create a feature branch: feature/fix-{page}-{description}
  3. Make the fix (use Claude to help: "Fix the spacing on the services hero to match the live site")
  4. Use Puppeteer / compare-styles to verify your fix matches
  5. Commit, push, create a PR targeting preview
The person who found the bug should NOT be the one to fix it. Josh found issues on Maddox's pages → Maddox fixes them (and vice versa). The tester re-verifies after the fix is merged. This is how real QA works: separation between testing and development.
4

Phase 4: Verify

After each PR is merged to preview:

  1. Wait for the deploy — preview.gptechadvisors.com auto-deploys on merge
  2. Re-test the specific section that was fixed
  3. Close the GitHub Issue if the fix looks correct
  4. If it's not right, reopen the issue with a comment explaining what's still off

When all issues are closed: do a full final pass across all pages. This is the last gate before we promote preview to production.

What "Done" Looks Like

  • Every page on preview.gptechadvisors.com visually matches gptechadvisors.com
  • The only differences are the intentional Contact Us removal
  • All GitHub Issues are closed
  • Final pass confirms no remaining differences at 1920px, 1400px, and 768px
  • Both of you sign off: "Looks good to me"

At that point, Stacey will create the final PR from preview to main and we go live.

Why This Matters

This isn't busywork. This is how professional software teams ship quality product:

  • Cross-testing catches blind spots. The builder always thinks it looks right.
  • Structured checklists prevent the "looks good to me" syndrome where everyone assumes someone else checked the details.
  • Using AI as a QA tool is a real-world skill. Companies are using exactly this approach to automate visual regression testing.
  • The SDLC loop (build → test → find bugs → fix → re-test → ship) is the foundation of every engineering team. You're doing the real thing.
